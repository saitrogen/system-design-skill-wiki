## What this is

Using S3 (or GCS, Azure Blob) as the primary storage layer for structured or semi-structured data, with compute engines (Athena, Spark, Snowflake, Databricks, DuckDB) reading from it. Not a traditional database — no native indexing, no row-level locking, no transactions. The model is: **cheap, durable, infinitely scalable storage + detached query compute**.

## When to use

- **Append-only and immutable data**: Logs, events, analytics facts, audit trails — write once, never update
- **Data lake and lakehouse architectures**: Snowflake and Databricks both use S3 (or equivalent) as their storage layer; compute scales independently from storage
- **Very large datasets at low cost**: S3 costs ~$0.023/GB/month vs. $0.10–0.50+/GB for managed RDBMS storage
- **Infrequent query patterns**: Data queried once a day in batch analytics jobs doesn't need a live database engine running 24/7
- **Multi-engine access**: Multiple compute engines (Spark, Athena, Presto, Snowflake) can all read from the same S3 bucket; enables decoupled architecture
- **Archive tier**: Offload historical data from [[postgresql]] or [[cassandra]] to S3 for cold reads

## When NOT to use

- **Low-latency reads**: S3 GET latency is 10–100ms+ per request — orders of magnitude slower than any in-memory or SSD-backed DB
- **Frequent small updates**: S3 has no in-place update; every "update" is a new object write. Updating one row in a 10GB Parquet file is expensive.
- **High-cardinality point lookups**: No indexes. Full object scan or partition-pruning only. Use a proper DB for `SELECT * WHERE id = 123`.
- **Transactional workloads**: No ACID guarantees natively. Open table formats (Apache Iceberg, Delta Lake) add transaction semantics on top, but with complexity.
- **Real-time serving**: Not the right layer for user-facing reads.

## Trade-offs

| Factor | Detail |
|--------|--------|
| Cost | Dirt cheap storage; pay for compute separately on query |
| Throughput | Very high for large sequential reads (Parquet column pruning) |
| Latency | High per-object; not suitable for OLTP |
| Indexing | None natively; Iceberg/Delta Lake add partition pruning and file-level statistics |
| Consistency | S3 provides strong read-after-write consistency (as of Dec 2020) |
| Durability | 11 nines (99.999999999%) — more durable than almost any self-managed DB |
| Updates | Object overwrite is atomic but full-object; no row-level update |
| Schema evolution | Parquet + Iceberg handle schema evolution gracefully |

**The open table format unlock**: Apache Iceberg and Delta Lake add ACID transactions, time travel, partition evolution, and hidden partitioning on top of raw S3. This is the foundation of the modern lakehouse. Snowflake and Databricks are effectively managed Iceberg/Delta query engines.

**Compute separation**: The storage-compute split means you pay for S3 continuously but spin up query compute only when needed — economically compelling for analytics workloads with bursty query patterns.

## Real-world examples

- **Snowflake**: Separates storage (S3/GCS/Azure Blob) from compute entirely; users query Parquet files via Snowflake's virtual warehouses
- **Databricks**: Delta Lake sits on S3; provides ACID transactions and time travel on object storage
- **Netflix**: Uses S3 as the primary data lake; Spark jobs process petabytes of data stored as Parquet/ORC
